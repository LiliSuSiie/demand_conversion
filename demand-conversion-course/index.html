💡 为什么要关心这个？

python pipeline_runner.py --config pipeline_config.json --preview

核心角色地图

每个优秀的软件项目就像一部电影，有不同的角色各司其职。在 Demand Conversion Pipeline 中，也有一群"演员"在幕后协作。

认识这些角色，能帮你在扩展功能时知道把新代码放在哪里，出问题时知道该检查哪个组件。

💡 为什么要关心角色分工？
就像修车要知道发动机、变速箱在哪一样。当你告诉 AI "在配置处理部分加个新功能" 或 "LLM 客户端这里有 bug"，你需要知道这些部分对应哪些代码文件。

主要演员阵容

🎭

指挥官 (pipeline_runner.py)

整个系统的总导演，负责协调所有其他角色按顺序工作

主要职责：
  • 解析命令行参数
  • 加载配置文件
  • 按顺序调用各个阶段
  • 处理错误和重试逻辑
⚙️

配置管家 (config.py)

管理所有的设置项，确保每个组件都知道自己该怎么工作

主要职责：
  • 读取和验证 JSON 配置
  • 提供默认值
  • 路径解析和验证
  • LLM 提供商配置
📖

文档阅读员 (requirement_parser.py)

专门负责理解 Word 文档，把人类的需求翻译成程序能处理的格式

主要职责：
  • 解析 Word 表格
  • 提取需求标题和描述
  • 处理空行和合并逻辑
  • 生成结构化需求对象
✍️

提示词作家 (generate_requirement_prompts.py)

把需求翻译成 AI 能理解的指令，就像给 AI 写剧本一样

主要职责：
  • 将需求分批组织
  • 生成标准化提示词模板
  • 添加输出格式说明
  • 保存为独立的提示词文件
🤖

AI 经纪人 (llm_client.py)

负责和各种 AI 服务商打交道，管理调用、重试、缓存等复杂逻辑

主要职责：
  • 管理不同 LLM 提供商
  • 处理并发调用和限流
  • 实现智能重试机制
  • 缓存管理和验证
🔍

用例挖掘师 (stages.py 中的解析逻辑)

从 AI 的自由文本回复中，用正则表达式精确提取出结构化的测试用例

主要职责：
  • 解析 AI 回复文本
  • 提取用例名称、操作、预期结果
  • 数据清洗和格式标准化
  • 生成最终测试用例对象

代码中的角色分工

代码这样组织：

pipeline_runner.py    # 指挥官
├── config.py          # 配置管家
├── requirement_parser.py  # 文档阅读员
├── generate_requirement_prompts.py  # 提示词作家
├── llm_client.py     # AI 经纪人
└── stages.py         # 用例挖掘师 + 整体流程

为什么这样分工：

  • 单一职责：每个文件只做一件事情，出错时容易定位
  • 可测试性：每个角色都可以独立测试
  • 可扩展性：想换 LLM 提供商？只改 llm_client.py
  • 可维护性：需求解析有问题？只看 requirement_parser.py
🎯 实用技巧
当你要求 AI 修改功能时，可以这样说："在 config.py 里添加新的配置项"、"修改 llm_client.py 的重试逻辑"。比说"改一下配置"、"修复 AI 调用"要精确得多。

👆 了解了角色分工，接下来看看配置是如何驱动整个系统行为的

配置如何驱动行为

想象一下，配置文件就像汽车的仪表盘——你通过调整不同的开关和数值，来控制整个系统的行为。

在 Demand Conversion Pipeline 中，几乎所有的行为都可以通过 pipeline_config.json 来控制，而不需要修改代码。

💡 为什么配置如此重要？
掌握配置，意味着你可以在不碰代码的情况下：切换 AI 模型、调整批处理大小、更改输出路径、开启调试模式。这是"零代码定制"的核心。

配置的层级结构

📁
pipeline_config.json
📂
paths
文件路径配置
🤖
llm
AI 模型配置
📦
batch
批处理配置
🚀
flags
执行开关配置

配置文件长这样：

{
  "paths": {
    "docx": "source/requirements.docx",
    "artifacts_dir": "artifacts",
    "output_excel": "output/test_cases.xlsx"
  },
  "llm": {
    "provider": "openai_compatible",
    "base_url": "https://yunwu.ai/v1",
    "model": "gpt-5.3-codex"
  },
  "batch": {
    "batch_size": 8
  },
  "flags": {
    "resume": false,
    "dry_run": false
  }
}

系统如何理解：

  • paths："去这些地方找文件，结果放这里"
  • llm："连这个 AI 服务，用这个模型"
  • batch："每次处理 8 个需求"
  • flags："要不要断点续传，要不要真正执行"

配置 vs 命令行：优先级游戏

系统有一套聪明的优先级规则：命令行参数 > 配置文件 > 默认值

最高优先级

命令行参数

python pipeline_runner.py --dry-run --resume

临时控制，这次运行特殊设置

⬇️
中等优先级

配置文件

"dry_run": false, "resume": false

项目的默认行为，长期稳定设置

⬇️
最低优先级

代码默认值

dry_run = False, resume = False

保底设置，防止配置缺失时程序崩溃

代码中的优先级逻辑：

def main() -> None:
    config = load_config(args.config)

    # 命令行覆盖配置文件
    if args.resume:
        config.flags.resume = True
    if args.dry_run:
        config.flags.dry_run = True
    if args.preview:
        config.flags.preview = True

实际效果：

即使配置文件里写着 "dry_run": false，只要你在命令行加了 --dry-run，系统就会按干跑模式执行。

这样既保证了配置的稳定性，又给了临时调整的灵活性。

👆 掌握了配置驱动，接下来深入了解 LLM 调用的奥秘

LLM调用与缓存恢复

调用 AI 就像点外卖：你不想每次都重复下单，也不想因为网络问题就前功尽弃。这个系统设计了一套精巧的缓存和重试机制。

理解这套机制，能帮你在开发 AI 应用时避免常见的坑：重复计费、超时失败、缓存污染。

💡 为什么要关心 LLM 调用细节？
AI 调用是昂贵且不可靠的。掌握缓存、重试、并发控制，能让你的 AI 应用既省钱又稳定。这些知识在任何 AI 项目中都通用。

调用前的安全检查：Preflight

就像飞行员起飞前要检查仪表一样，系统在批量调用前会先做一次预检查。

✅

1. 连通性检查

能连上 API 服务器吗？

✅

2. 认证检查

API Key 有效吗？

✅

3. 响应格式检查

返回的内容是预期格式吗？

✅

4. 空响应检查

模型会返回有意义的内容吗？

Preflight 代码逻辑：

async def _run_preflight(self) -> None:
    # 构造测试请求
    test_request = LLMRequest(
        prompt="test preflight",
        metadata={"chunk_id": "preflight"}
    )

    # 尝试调用一次
    response = await self._send_single_request(test_request)

    # 检查返回是否有效
    if not response.text.strip():
        raise ValueError("LLM returned empty response")

为什么需要 Preflight：

  • 快速失败：配置错误时立即发现，不浪费时间
  • 批量保护：避免 50 个请求都失败，浪费配额
  • 调试友好：清晰的错误信息，快速定位问题

并发控制：不被限流封号

大部分 AI 服务都有速率限制。系统用信号量控制并发，避免触发限流。

3

等待队列

请求 #4
请求 #5
请求 #6

正在执行

请求 #1
请求 #2
请求 #3

已完成

并发控制代码：

async def send_batch(self, requests: List[LLMRequest]) -> List[LLMResponse]:
    # 创建信号量，限制并发数
    semaphore = asyncio.Semaphore(self.config.max_concurrency)

    async def _send_with_semaphore(request):
        async with semaphore:  # 获取信号量
            return await self._send_single_request(request)

    # 并发执行，但受信号量限制
    tasks = [_send_with_semaphore(req) for req in requests]
    return await asyncio.gather(*tasks)

为什么要限制并发：

  • 避免限流：太多并发请求会触发 API 的限流机制
  • 资源保护：避免过载本地网络和内存
  • 友好使用：避免被 API 提供商标记为滥用

重试机制：网络不稳定时的救星

第1次尝试
❌ 网络超时
等待 5 秒
第2次尝试
❌ 服务器错误
等待 10 秒
第3次尝试
✅ 调用成功

重试策略：

  • 指数退避：5秒 → 10秒 → 20秒，避免加剧服务器压力
  • 有限次数：最多重试3次，避免死循环
  • 区分错误：网络错误重试，认证错误直接失败
  • 缓存保护：重试成功后立即缓存，避免再次失败

测试你的理解

系统遇到相同的提示词时会发生什么？

正确！ 系统会计算提示词的哈希值，相同内容对应相同哈希。如果 prompt_cache/chunk_X_hash.json 文件存在且有效，就直接使用缓存结果，这样既节省了成本，又提高了响应速度。

👆 理解了 LLM 调用机制，接下来看看如何从 AI 的自由文本中提取结构化数据

从文本到测试用例

AI 模型回复的是自然语言文本，但我们需要的是结构化的测试用例。这个转换过程就像从一堆散装零件中，按照图纸组装出精密的机械。

掌握这个过程，能帮你在处理任何 AI 输出时，都知道如何提取有用的信息。

💡 为什么要关心文本解析？
AI 的输出往往是"接近正确"的文本，如何从中稳定提取结构化数据，决定了你的 AI 应用的可靠性。这是所有 AI 应用的核心挑战。

AI 输出的典型格式

系统要求 AI 按照特定格式回复，但实际输出往往有细微差异。

理想输出：

• 用例：登录成功 - 操作：输入正确用户名和密码点击登录 - 预期：跳转到主页面显示欢迎信息
• 用例：登录失败 - 操作：输入错误密码点击登录 - 预期：显示密码错误提示信息
✅ 完美格式，直接解析

变体1（空格差异）：

•用例：登录成功-操作：输入正确用户名和密码点击登录-预期：跳转到主页面显示欢迎信息
• 用例：登录失败 -  操作：输入错误密码点击登录  -  预期：显示密码错误提示信息
⚠️ 空格不一致，需要容错

变体2（符号差异）：

- 用例：登录成功 - 操作：输入正确用户名和密码点击登录 - 预期：跳转到主页面显示欢迎信息

变体3（格式错误）：

预期结果：跳转到主页面显示欢迎信息

操作：输入错误密码点击登录
预期：显示密码错误提示信息
❌ 格式完全不同，解析失败

正则表达式的威力

系统使用正则表达式来从文本中提取结构化信息。

解析规则拆解：

^[•*-]\s*用例：(?P<name>.+?)\s*-\s*操作：(?P<operation>.+?)\s*-\s*预期：(?P<expected>.+?)$
^[•*-]\s*
行开头的列表符号 + 可选空格
用例：(?P<name>.+?)\s*
"用例："后面的内容保存为 name
-\s*操作：(?P<operation>.+?)\s*
"-操作："后面的内容保存为 operation
-\s*预期：(?P<expected>.+?)$
"-预期："后面的内容保存为 expected

匹配演示：

输入文本： 
• 用例：登录成功 - 操作：输入正确用户名和密码点击登录 - 预期：跳转到主页面显示欢迎信息
提取结果：
name: "登录成功"
operation: "输入正确用户名和密码点击登录"
expected: "跳转到主页面显示欢迎信息"

解析核心代码：

LLM_CONVERT_REGEX = re.compile(
    r"^[•*-]\s*用例：(?P<name>.+?)\s*-\s*操作：(?P<operation>.+?)\s*-\s*预期：(?P<expected>.+?)$",
    re.MULTILINE
)

def parse_test_cases(text: str) -> List[TestCase]:
    cases = []
    for match in LLM_CONVERT_REGEX.finditer(text):
        case = TestCase(
            name=match.group('name').strip(),
            operation=match.group('operation').strip(),
            expected=match.group('expected').strip()
        )
        cases.append(case)
    return cases

关键设计考量：

  • 容错性：\s* 匹配可选空格，应对格式差异
  • 贪婪控制：.+? 非贪婪匹配，避免跨字段错误
  • 清理：strip() 去除首尾空格
  • 多行模式：re.MULTILINE 支持每行独立匹配

常见问题诊断手册

配置文件错误

症状：程序启动时就报错，提示配置相关问题
诊断：
  • 检查 JSON 语法是否正确（逗号、括号、引号）
  • 验证必填字段是否存在
  • 确认文件路径是否存在
  • 检查 API key 环境变量是否设置
解决： python -c "import json; print(json.load(open('pipeline_config.json')))"

LLM 调用失败

症状：在第三阶段停止，提示网络错误或API错误
诊断：
  • 检查网络连接和防火墙
  • 验证 API key 有效性
  • 确认 base_url 和 model 配置正确
  • 检查是否超出配额或触发限流
解决： 查看 prompt_cache/ 目录，检查已缓存的请求，使用 --resume 从失败处继续

解析失败

症状：LLM 调用成功，但 parsed_cases.json 中用例数量很少
诊断：
  • 查看 responses/chunk_*.txt 检查 AI 输出格式
  • 检查正则表达式是否匹配实际输出
  • 确认 AI 是否按要求输出
  • 检查是否有非标准格式的用例
解决： 优化提示词，或扩展正则表达式支持更多格式变体

文件访问问题

症状：提示文件不存在或权限不足
诊断：
  • 确认输入的 Word 文件路径正确
  • 检查输出目录是否有写入权限
  • 验证相对路径是否正确
  • 确认文件没有被其他程序占用
解决： 使用绝对路径，或调整目录权限，关闭占用文件的程序

性能优化建议

🔄

缓存策略

📦

批次大小

调整 batch_size：太小浪费请求，太大降低质量，建议 5-10。

提示词质量

高质量的提示词能显著提升解析成功率，减少重试次数。
