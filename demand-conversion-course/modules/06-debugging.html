这些技能不仅适用于当前系统，在你使用 AI 构建其他应用时同样有用。

常见问题诊断手册

配置文件错误

症状：程序启动时就报错，提示配置相关问题
诊断：
  • 检查 JSON 语法是否正确（逗号、括号、引号）
  • 验证必填字段是否存在
  • 确认文件路径是否存在
  • 检查 API key 环境变量是否设置
解决： python -c "import json; print(json.load(open('pipeline_config.json')))"

LLM 调用失败

症状：在第三阶段停止，提示网络错误或API错误
诊断：
  • 检查网络连接和防火墙
  • 验证 API key 有效性
  • 确认 base_url 和 model 配置正确
  • 检查是否超出配额或触发限流
解决： 查看 prompt_cache/ 目录，检查已缓存的请求，使用 --resume 从失败处继续

解析失败

症状：LLM 调用成功，但 parsed_cases.json 中用例数量很少
诊断：
  • 查看 responses/chunk_*.txt 检查 AI 输出格式
  • 检查正则表达式是否匹配实际输出
  • 确认 AI 是否按要求输出
  • 检查是否有非标准格式的用例
解决： 优化提示词，或扩展正则表达式支持更多格式变体

文件访问问题

症状：提示文件不存在或权限不足
诊断：
  • 确认输入的 Word 文件路径正确
  • 检查输出目录是否有写入权限
  • 验证相对路径是否正确
  • 确认文件没有被其他程序占用
解决： 使用绝对路径，或调整目录权限，关闭占用文件的程序

性能优化建议

⚡

并发调优

根据 API 限制调整 max_concurrency。太大触发限流，太小速度慢。

🔄

缓存策略

相同项目的迭代开发中，大部分提示词会重复，充分利用缓存机制。

📦

批次大小

调整 batch_size：太小浪费请求，太大降低质量，建议 5-10。

🎯

提示词质量

高质量的提示词能显著提升解析成功率，减少重试次数。

测试你的掌握程度

当系统在第三阶段失败，但你看到 `prompt_cache/` 目录下已经有一些缓存文件时，最好的策略是什么？

正确！ 已有的缓存代表成功的 LLM 调用，删除它们会浪费时间和金钱。使用 --resume 能让系统自动识别已完成的部分，只重新执行失败的请求。这正是断点恢复机制的价值所在。
