配置如何驱动行为

想象一下，配置文件就像汽车的仪表盘——你通过调整不同的开关和数值，来控制整个系统的行为。

在 Demand Conversion Pipeline 中，几乎所有的行为都可以通过 pipeline_config.json 来控制，而不需要修改代码。

💡 为什么配置如此重要？
掌握配置，意味着你可以在不碰代码的情况下：切换 AI 模型、调整批处理大小、更改输出路径、开启调试模式。这是"零代码定制"的核心。

配置的层级结构

📁
pipeline_config.json
📂
paths
文件路径配置
🤖
llm
AI 模型配置
📦
batch
批处理配置
🚀
flags
执行开关配置

配置文件长这样：

{
  "paths": {
    "docx": "source/requirements.docx",
    "artifacts_dir": "artifacts",
    "output_excel": "output/test_cases.xlsx"
  },
  "llm": {
    "provider": "openai_compatible",
    "base_url": "https://yunwu.ai/v1",
    "model": "gpt-5.3-codex"
  },
  "batch": {
    "batch_size": 8
  },
  "flags": {
    "resume": false,
    "dry_run": false
  }
}

系统如何理解：

  • paths："去这些地方找文件，结果放这里"
  • llm："连这个 AI 服务，用这个模型"
  • batch："每次处理 8 个需求"
  • flags："要不要断点续传，要不要真正执行"

配置 vs 命令行：优先级游戏

系统有一套聪明的优先级规则：命令行参数 > 配置文件 > 默认值

最高优先级

命令行参数

python pipeline_runner.py --dry-run --resume

临时控制，这次运行特殊设置

⬇️
中等优先级

配置文件

"dry_run": false, "resume": false

项目的默认行为，长期稳定设置

⬇️
最低优先级

代码默认值

dry_run = False, resume = False

保底设置，防止配置缺失时程序崩溃

代码中的优先级逻辑：

def main() -> None:
    config = load_config(args.config)

    # 命令行覆盖配置文件
    if args.resume:
        config.flags.resume = True
    if args.dry_run:
        config.flags.dry_run = True
    if args.preview:
        config.flags.preview = True

实际效果：

即使配置文件里写着 "dry_run": false，只要你在命令行加了 --dry-run，系统就会按干跑模式执行。

这样既保证了配置的稳定性，又给了临时调整的灵活性。

LLM 提供商的配置魔法

系统支持多种 AI 服务商，通过 provider 字段自动选择对应的接口实现。

{
  "provider": "openai_compatible",
  "base_url": "https://yunwu.ai/v1",
  "model": "gpt-5.3-codex",
  "options": {
    "endpoint": "responses"
  }
}
{
  "provider": "custom_http",
  "base_url": "https://your-api.com",
  "model": "testcase-v1",
  "options": {
    "method": "POST",
    "path": "/generate",
    "request_format": "prompt_model"
  }
}
{
  "provider": "anthropic",
  "model": "claude-3-sonnet",
  "api_key_env": "ANTHROPIC_API_KEY"
}

同样的代码，不同的配置，就能对接完全不同的 AI 服务。这就是策略模式的威力。

配置驱动的典型场景

🔧 开发调试

{
  "flags": {
    "dry_run": true,
    "preview": true
  },
  "batch": {"batch_size": 2}
}

不调用真实 LLM，快速验证流程，小批量测试

🚀 生产运行

{
  "flags": {
    "dry_run": false,
    "resume": true
  },
  "batch": {"batch_size": 8},
  "llm": {"max_concurrency": 3}
}

真实执行，支持断点续传，高效并发

🧪 实验新模型

{
  "llm": {
    "model": "gpt-4-turbo",
    "base_url": "https://new-api.com/v1"
  },
  "paths": {
    "artifacts_dir": "experiments/gpt4"
  }
}

切换模型，独立输出目录，对比效果

测试你的理解

假设配置文件里设置了 `"dry_run": false"`，但你运行命令时加了 `--dry-run` 参数，最终会发生什么？

正确！ 命令行参数的优先级最高。代码中有明确的逻辑：if args.dry_run: config.flags.dry_run = True，会覆盖配置文件的设置。这种设计让你可以灵活地进行临时调整。
