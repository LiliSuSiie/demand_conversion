一键运行时发生了什么

想象一下：你有一个 Word 文档里写着密密麻麻的需求，然后只需要一条命令，它就自动变成了整齐的 Excel 测试用例表格。

这听起来像魔法，但其实是一个精心设计的数据流水线在工作。

💡 为什么要关心这个？
理解数据怎么流动，能帮你：调试问题时知道在哪个环节卡住了，向 AI 要求类似功能时能说清楚架构，扩展功能时不会搞乱现有流程。

代码这样说：

python pipeline_runner.py --config pipeline_config.json --preview

人话翻译：

运行完整流水线，并额外写出需求解析预览文件。注意：--preview 不会跳过 LLM；只想检查 Word 解析和 prompt 生成时，要加 --dry-run。

需求块 11

每一步都有断点保护，所以即使中途出错，也能接着上次的进度继续。

实际命令运行时

用户: python pipeline_runner.py --config pipeline_config.json
Pipeline: 好的，开始检查配置文件...
Pipeline: 配置正常！开始第一阶段：解析 Word 文档
文档解析器: 找到 Word 文件，优先识别需求表格；如果没有有效表格，就切到段落标题解析...
文档解析器: 解析完成！v1.9.0 文档发现 11 个需求块
Pipeline: 第二阶段：生成 AI 提示词
提示词生成器: 按每批 5 个需求分组，生成 3 个提示词文件
Pipeline: 第三阶段：调用 LLM
LLM 客户端: 连接到 openai_compatible 服务，使用 claude-opus-4-1-20250805，开始批量处理...
LLM 客户端: 所有批次处理完成！
Pipeline: 第四阶段：解析测试用例
用例解析器: 从 AI 回复中提取出 141 条测试用例
Pipeline: 第五阶段：导出 Excel
Excel 导出器: ✅ 导出完成：output/output_test_cases.xlsx

测试你的理解

如果程序在第三阶段（LLM 调用）时网络断了，重新运行会怎样？

正确！ 使用 --resume 时，Pipeline 会检查 artifacts/pipeline_state.json，跳过已完成阶段；LLM 层还会读取 prompt_cache/chunk_*.json，避免重复调用已缓存的批次。
